Linux Kernel Deduplication Patch Study: Proposal and Data Exploration

2026-06-30

Summary

In this post, we describe the plan for a small empirical study about code deduplication patches in the Linux kernel mailing lists. The goal is to understand how often kernel patches are related to duplicate or redundant code, how these patches are discussed by maintainers, and whether accepted and rejected patches show different patterns.

Our immediate objective is to define a reliable filtering pipeline, manually validate a subset of threads, use a language model to scale the annotation process, and connect the resulting dataset with Patchwork metadata in order to estimate whether each patch was accepted, rejected, or left unresolved.

Motivation

Code duplication is usually discussed as a software maintenance problem: duplicated logic can make bugs easier to copy, fixes harder to apply consistently, and codebases more difficult to evolve. However, in a large project such as the Linux kernel, not every repeated pattern is obviously bad. Sometimes duplication is intentionally kept for readability, hardware-specific behavior, subsystem isolation, or because an abstraction would make the code harder to understand.

This makes kernel patch discussions a useful source of evidence. Instead of only asking whether duplicate code exists, we can study how developers justify removing it, when maintainers accept deduplication patches, and when they push back against them.

Research questions

We plan to organize the study around four main questions.

Question Planned measurement
How many threads are related to code duplication or deduplication? Count threads selected by a permissive regex filter and then validated manually or with an LLM.
Among those patches, how many were accepted or rejected? Cross the selected threads with Patchwork data and assign an acceptance status when available.
Do maintainers differ in how they respond to deduplication patches? Compare acceptance and rejection rates by maintainer, using either the last maintainer reply or all maintainer replies in a thread.

Defining duplication

The first step is to define what we mean by duplication. We will start from the classic clone types and adapt them to the kernel patch setting. In this initial plan, we consider clone types 1, 2, 3, and 4, but we also include simpler deduplication changes that may not fit cleanly into the usual taxonomy.

Type-1 clones are near-identical code fragments, usually differing only in whitespace, comments, or formatting. Type-2 clones also allow changes in names, literals, or identifiers. Type-3 clones include copied code with added, removed, or modified statements. Type-4 clones are semantically similar pieces of code that implement the same behavior but may look structurally different.

In addition to these standard categories, kernel patches often include practical deduplication changes that are more local and idiomatic. For example, a patch may remove a repeated null check because a helper function is already null-aware, replace repeated logic with an existing helper, or simplify two branches that perform the same operation. These changes are important for our study even when they are not presented as traditional clone removal.

Manual validation

The next step is to separate a test set of candidate threads and inspect them manually. For each thread, we will decide whether it is actually about duplication or deduplication, and if so, which kind of duplication it represents.

Manual validation has two goals. First, it gives us an estimate of how noisy the permissive regex filter is. Second, it creates a reference set that can be used to evaluate whether an LLM can reproduce our labels.

Thread Manual label Duplication type Notes
thread-id-1 relevant / not relevant type 1 / type 2 / type 3 / type 4 / local deduplication short explanation
thread-id-2 relevant / not relevant type 1 / type 2 / type 3 / type 4 / local deduplication short explanation

We will also keep examples of diffs for each duplication type. These examples will be useful both for the paper proposal and for prompting the LLM. The goal is not only to classify threads, but also to make the categories concrete and reproducible.

LLM-assisted filtering

Once we have a manually labeled validation set, we will pass the same threads to an LLM and compare its labels with ours. The LLM will receive the subject, commit message, relevant email body, and possibly the patch diff. It will then answer whether the thread is about duplication and, when relevant, assign a duplication category.

If the LLM agrees with the manual labels on the validation set, we will use it to classify the remaining candidate threads. If the agreement is weak, we will revise the prompt, improve the category definitions, or reduce the LLM task to a simpler binary decision: duplication-related or not. 

Planned LLM outputs:
{
  "is_deduplication_related": true,
  "duplication_type": "local_deduplication",
  "confidence": "high",
  "evidence": "The patch removes a repeated check because the helper already handles NULL."
}

This step is especially useful because many relevant patches may not use exactly the same vocabulary. A regex can find obvious terms, but an LLM may be better at recognizing cases where the patch is about consolidating logic, removing repeated checks, or replacing duplicated behavior with a helper function.

Accepted and rejected patches

To label patches as accepted or rejected, we plan to cross the selected threads with Patchwork data from patchwork.kernel.org. When a thread corresponds to a patch tracked by Patchwork, we can use its available status as an external signal of whether the patch was accepted, rejected, superseded, or left in another state.

This label will likely be imperfect. Patch acceptance is not always a simple binary variable: a patch may be revised in a later version, accepted as part of a series, superseded by another implementation, or discussed without a clear final status. For that reason, we will keep the raw Patchwork status when possible and only collapse it into accepted/rejected categories for analyses where this simplification is appropriate.

Comparing maintainers

A central part of the study is to compare how maintainers respond to deduplication patches. The first plan is to identify, for each thread, the last maintainer who replied. This gives a simple proxy for the maintainer most closely associated with the final direction of the discussion.

However, this may be too simplistic. If one maintainer frequently replies last across many threads, the analysis could overrepresent that person. Therefore, we also keep a second plan: instead of considering only the last maintainer reply, we can collect all maintainers who participated in the thread and compare acceptance or rejection patterns across that broader set.

Analysis Definition Risk
Plan A Use the last maintainer who replied in each thread. May be biased if one maintainer often replies last.
Plan B Use all maintainers who replied in each thread. May overcount threads with long discussions or many maintainers.

We will treat maintainer comparison as exploratory rather than causal. A higher or lower acceptance rate for a maintainer does not necessarily mean that the maintainer is more or less favorable to deduplication. It may reflect subsystem differences, patch quality, contributor experience, or the kinds of deduplication changes submitted to that maintainer.
